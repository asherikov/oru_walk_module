00099  * @section pControlLoop Control loop
00100  * A sequence diagram of one control loop:
00101   @msc
00102     dcm [label="DCM thread"], cb [label="DCM callback"], ct [label="Control thread"];
00103     dcm =>> cb [label = "Execute"];
00104     cb  =>  cb [label = "NOOP"];
00105     dcm =>> cb [label = "Execute"];
00106     cb  ->  ct [label = "Activate"];
00107     ct  =>  ct [label = "Feedback error"];
00108     ct  =>  ct [label = "Solve MPC"];
00109     ct  =>  ct [label = "Solve IK (1)"];
00110     ct  =>  ct [label = "Send joint angles (1)"];
00111     ct  =>  ct [label = "Solve IK (2)"];
00112     ct  =>  ct [label = "Send joint angles (2)"];
00113     ---        [ label = "Repeat until walk is finished"];
00114   @endmsc
00115  * The interconnection between DCM thread, DCM callback and control thread is explained
00116  * in section @ref pControlThread, this section also describes the parameters of the
00117  * control thread.
00118  *
00119  * Operations displayed on this diagram are described below.
00120  * \n\n
00121  *
00122  *
00123  * @section pControlThread Control thread
00124  *
00125  * We use a separate thread to control a robot, this thread is synchronized
00126  * with DCM. The control sampling time using in this thread is 20 ms. The
00127  * thread is spawned when the 'walk' command is received by the module.
00128  *
00129  * For some time we have been using hook (callback) to ALMotion module. The 
00130  * main reason why we switched to a separate thread was the lack of 
00131  * synchronization between DCM and ALMotion threads. This is important since 
00132  * DCM updates joint angles in the memory, and there is no other way to get 
00133  * the values of the angles.
00134  *
00135  * This is probably not a good solution, which would not work as well in future
00136  * releases of firmware.
00137  *
00138  * @subsection pDCMsync Synchronization with DCM
00139  * The DCM is activated each 10 ms. We register a callback function, which is
00140  * called when DCM finishes its work. Thus the callback function is also
00141  * executed each 10ms. The callback has a counter of executions, on each even
00142  * execution (i.e. each 20 ms.) it wakes up the control thread using conditional
00143  * variable -- a synchronization tool provided by the Boost.
00144  *
00145  * @subsection pThreadPriority Priority of the control thread
00146  * The priority of the control thread plays a critical role, since sometimes
00147  * it is not possible to complete all necessary computations in 20ms. using 
00148  * default priority. 
00149  *
00150  * We use SCHED_FIFO scheduling policy, the same as DCM thread uses. The priority
00151  * is set to 65. Note, that NaoQi does not run with root privileges and it limits
00152  * the possible changes of priorities. 
00153  * \n\n
00154  * 
00155  *
00156  * @section pFeedback Feedback
00157  * There are two types of feedback. 
00158  *
00159  * The first one is shown on sequence diagram above. The position of center of mass, that
00160  * is used as the initial position in the MPC solver, is corrected using the position
00161  * of center of mass computed from the sensor data. The feedback is applied only if the
00162  * difference between these positions is bigger than predefined threshold. Also, the error
00163  * is multiplied by a gain (< 1.0) before addition.
00164  *
00165  * The second one is required to avoid jumps of center of mass, when the support foot is 
00166  * changed. In this case the position of new support foot is not the same as expected due
00167  * to imprecision of movements. Therefore, we have to find the new position using sensors
00168  * and change this position in the footstep pattern. It is also necessary to synchronize
00169  * joint angles in the model of the robot with the joint angles obtained from sensors.
00170  * \n\n
00171  *
00172  *
00173  * @section pIGM Inverse kinematics
00174  * We compute inverse kinematics and set joint angles twice in one control loop. 
00175  *
00176  * The first computation produces the joint angles that must be reached in one
00177  * control sampling period, i.e. when the control thread is activated again. 
00178  * Since these joint angles are reached when the control loop is activated next 
00179  * time, the controllers have to have some commands to execute while the control 
00180  * loop works. The second solution is produced to keep controllers busy, it prevents 
00181  * jerky motions. The joint angles obtained in the second computation are expected 
00182  * to be reached in two control sampling periods. 
00183  * \n\n
00184  */

00187 /**
00188  * @page pNotes Notes
00189  *
00190  * @section pNotesDoc Exerpts from the documentation
00191  * - LHipYawPitch and RHipYawPitch share the same motor so they move 
00192  *   simultaneously and symmetrically. In case of conflicting orders
00193  *   LHipYawPitch always takes the priority.
00194  *
00195  *
00196  * @section pNotesGen General
00197  * - A robot cannot execute the desired foot trajectories precisely. 
00198  *   The errors may be very different for the right and left foot.
00199  *
00200  * - If the sensor input from a joint is used as a command for this joint
00201  *   in a callback function, the joint becomes compliant. 
00202  *
00203  * - The callback function that controls the robot, can be hooked to DCM
00204  *   module or ALMotion module, in the first case it would be called each 
00205  *   10ms, in the second - 20 ms.
00206  *
00207  * - The DCM and ALMotion threads are not synchronized.
00208  *
00209  * - When the module is running on PC in simulation, RHipYawPitch is not
00210  *   updated and is always equal to 0.0. We haven't checked it on th robot
00211  *   but it is likely to behave the same way.
00212  *
00213  *
00214  *
00215  * @section pNotesOur Our module
00216  * - In the current version we switch between the left and the right 
00217  *   foot. We need to choose one foot as a reference to find a position 
00218  *   of CoM. In theory, it is possible to use the same foot, but when 
00219  *   this foot is lifted it's real position can be found only using the 
00220  *   other foot as a reference due to errors. A big error in foot position 
00221  *   will lead to a big error in CoM position.
00222  *
00223  * - Since we cannot follow the desired feet trajectories precisely (the real
00224  *   positions are lower), there is a significant error in CoM position along
00225  *   Z and Y axes.
00226  *
00227  *
00228  * @section pNotesMPC MPC solver
00229  * - The length of the preview window affects the ability of the robot to
00230  *   compensate for disturbances. On the other hand a long preview window 
00231  *   increases the number of constraints in the MPC problem. 
00232  *
00233  * - Number of active constraints has strong impact on performance. Under 
00234  *   strong disturbances the number of activated constraints in the MPC 
00235  *   makes it impossible to solve the problem in the available time.
00236  *
00237  *
00238  * @section pNotesIGM IGM
00239  * - When the orientaion of the torso is fixed (x and y axes), the hip roll
00240  *   joints are highly loaded and may hit their limits depending on the MPC
00241  *   gains.
